As said above, the options list can contain any kind of item, and this is handy since selecting an option will commit the actual item to the data model, allowing to store complex objects.

But the requirement is to be able to display those options, be it in the list of the popup or inside the input field.

String representation for the options list: resultFormatter

The option item can be displayed from a string. In this case, use the property resultFormatter to pass a function which takes the option item (whose model is up to you) as input and returns a string as output.

Complex representation for the options list: resultTemplate

Sometimes displaying a simple string is not enough. In that case, you can use a template by using the property resultFormatter, which expects a template reference.

The template receives in its context:

  • the option item, as result
  • the input field content (the search term), as term
String representation for the input field: inputFormatter

It is also possible to customize the display of the option inside the input field, using the property inputFormatter which expects a function taking the option item as input and returning a string as output.

There are two limitations though:

  • one is technical: since this is an input element, only a string representation can be used, no custom template
  • one is from the design of the widget: since the input field is also used to edit the search term, altering the content of the field will change the initial value you edit for your next search. Therefore this is more suitable for conversion from complex object to a string, or for simple transformations (like changing case) which would lead to a value that can still be handled by the search algorithm in use.
